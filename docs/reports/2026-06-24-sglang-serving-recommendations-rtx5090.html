SGLang Serving Recommendations — RLM Inference on RTX 5090

Model: gemma-4-12B (Gnosis-MedPolicy-13, v13) — single RTX 5090 / SM_120 / WSL2

Date: 2026-06-24  ·  Engineering advisory

Status: advisory — ADR-0016 remains PROPOSED, not accepted. Do not flip serving defaults until the GATE #2 accuracy A/B clears.

Executive Summary

Three serving decisions are on the table for the RLM inference server, and the verdicts diverge sharply. KV-cache quantization stays on fp8_e4m3 in production but is only an amber keep: its effect on long-context retrieval accuracy is unvalidated and is a prime suspect in the GATE #2 accuracy gap, so it must be A/B'd against bf16 KV. Speculative decoding (DFlash) is a flat NO-GO: it structurally zeroes out our dominant latency lever (RadixAttention prefix reuse) and targets a low-batch decode regime we never enter. Config tuning is the green, actionable win: size everything around one invariant so the orchestrator prefix is never evicted. The single cross-cutting priority above all others is to run the GATE #2 accuracy A/B before promoting any default — the stack's gate is accuracy (~0 correct on small samples), not latency, and three suspects (fp8 KV precision, the gemma4 chat-template/thinking-mode, and small-sample variance) must be disentangled first.

Verdict at a Glance

Topic Verdict One-line recommendation
KV quantization Validate first Keep fp8_e4m3 KV in production, but A/B it against bf16 KV on int4-ct weights before declaring it accuracy-safe.
Speculative decoding (DFlash) No-go Leave speculative decoding OFF — it kills prefix-cache reuse (#40624) and helps a regime we never reach.
Config tuning Adopt Keep slots * subcall_context_limit at ~½ the pool so RadixAttention never evicts the orchestrator prefix.

Use-case context

The RLM pattern: an inference engine/orchestrator writes programs in a REPL that slice, search, and recursively query a large offloaded context (a ~314k-char multihop needle-in-haystack), driving co-batched map-reduce sub-calls. The workload is prefill-heavy and prefix-cache-heavy — a long shared system-prompt prefix plus the same context chunks reused across hops, so RadixAttention prefix reuse is the dominant latency lever. Hardware is one RTX 5090 (32 GB, Blackwell consumer SM_120) under WSL2. The model is gemma-4-12B (gemma4_unified, text-only), served on :8080.

Current baseline config:

--quantization compressed-tensors      # W4A16-g32-CT
--kv-cache-dtype fp8_e4m3
--attention-backend triton
--mem-fraction-static 0.80
--context-length 32768
--cuda-graph-max-bs 24
--tp 1

Topic 1 — KV-Cache Quantization on the RTX 5090

Amber — keep, but unvalidated fp8_e4m3 KV is currently mandatory for capacity, not chosen for accuracy. Its effect on long-context retrieval is genuinely untested and is one of three candidate root-causes of the GATE #2 accuracy failure. It must be A/B'd against bf16 KV before it is declared safe.

Recommendation

Keep --kv-cache-dtype fp8_e4m3 as the production KV dtype, but treat its retrieval-accuracy effect as UNVALIDATED — it is the open variable in the GATE #2 accuracy gap and must be A/B'd against bf16 KV (now that int4-ct weights free the pool) before it is declared safe. Do NOT chase q8_0 KV or TurboQuant.

Why (grounded in our findings)

fp8 KV is currently mandatory for capacity, not accuracy. bf16 weights (24.6 GB) left max_total_num_tokens=194 (a KV-starved scheduler spin); fp8 weights (12.8 GB) plus fp8_e4m3 KV gave a ~98k-token pool (project_sglang-v13-serving-gate1).

The KV-dtype choice is now decoupled from pool pressure by int4-ct weights (~8.5 GB, the canonical -w4a16-g32-ct /7.5 Google-exact scheme, project_int4-quant-v13-w4a16). With weights that small, bf16 KV becomes affordable — which is precisely what enables the test of whether fp8 KV costs retrieval accuracy.

fp8 KV precision is explicitly one of three candidate root-causes of the GATE #2 accuracy failure (ADR-0016 §GATE #2 item b; project_sglang-radixattention-pool-tuning): plain-multihop scored ~0 correct, and lossy KV on haystack tokens can blur the needle in ~314k-char retrieval.

q8_0 KV is a llama.cpp GGML format, NOT an SGLang --kv-cache-dtype — the right SGLang A/B is fp8_e4m3 vs bf16 KV; llama q8_0 is the accuracy baseline-to-beat on a different engine. TurboQuant is correctly iceboxed (project_turboquant-iceboxed): 4-bit KV only wins when memory is THE binding constraint (we are not, on the fp8 hybrid-SWA pool: ~105k SWA + ~131k full), and TQ taxes prefill (WHT rotation + quantize + pack, 56–75% of bf16, worse with length) — a direct hit on our prefill-heavy profile. vLLM's study found FP8 dominates the Pareto frontier; TQ k8v4 gives no advantage over FP8.

Concrete config / values

Keep the current int4-ct baseline:

--quantization compressed-tensors
--kv-cache-dtype fp8_e4m3
--attention-backend triton
--mem-fraction-static 0.80
--cuda-graph-max-bs 24
--context-length 32768
--tp 1

Run a 3-arm KV-dtype A/B on int4-ct weights (weights fixed, KV isolated):

Arm A  --kv-cache-dtype fp8_e4m3   # incumbent
Arm B  --kv-cache-dtype auto       # bf16/fp16 KV, now feasible
                                  # pair with reduced subcall_context_limit
                                  # so eviction does not confound
Arm C  --kv-cache-dtype fp8_e5m2   # only if A loses to B

Use the canonical -w4a16-g32-ct (/7.5 Google-exact) checkpoint, NOT the AutoRound /8 one (different scale schemes). Keep mem-fraction-static 0.80 + cuda-graph-max-bs 24 (validated live 2026-06-25: with context-length 32768 bounding the SWA pool to ~96k tokens, the full decode-graph set bs[1..24] captures in ~132 s using only 0.05 GB — there is no capture-OOM pressure, so use the wider bs24 over bs16 for more decode-graph coverage).

Risks & unknowns

  • fp8-KV accuracy effect on long-context retrieval is genuinely UNTESTED — the headline unknown.
  • SM_120 caveat: only --attention-backend triton is viable on gemma-4/SM_120 (flashinfer gemma4-forbidden; trtllm_mha ValueErrors on SM_120, #14814). FP8 block-wise had SM_120 gaps (#9233) but our per-token fp8_e4m3 KV works — a watch-item, not a blocker.
  • Confounds: the GATE #2 gap also has two other suspects (small-sample variance; chat-template / thinking-mode under auto reasoning_parser=gemma4) — the KV A/B must control these.
  • A pre-quantized fp8 checkpoint on disk (vs ~17s online re-quant) does NOT change the KV-dtype choice — it only speeds startup.

What to measure

  • PRIMARY: multihop retrieval accuracy (correct-answer rate) on the plain-multihop subset, fp8 vs bf16 KV, on int4-ct weights, memory off first then on.
  • Pool size: max_total_num_tokens per arm.
  • Per-Prefill-batch #cached-token vs #new-token from the server LOG (NOT the cache_hit_rate Prometheus gauge, which reads 0.000 when idle).
  • Latency (s/iter) as a tiebreaker only — accuracy is the gate.

Topic 2 — Speculative Decoding (DFlash)

Red — NO-GO Do not enable DFlash / EAGLE-style speculative decoding for this workload. Two independent disqualifiers: (1) it structurally fights our single most important lever — DFlash + gemma-4 hybrid attention gives ~0% prefix-cache hits (#40624); and (2) spec-decode helps low-batch decode, while our workload is prefill-bound and batch-saturated, so the regime where it pays never occurs.

Recommendation

NO-GO. Leave speculative decoding OFF. The two disqualifiers above are independent — either alone is sufficient, and together they make a net win effectively impossible for this workload.

Why (grounded in our findings)

The prefix-cache conflict is decisive for US specifically. Recorded research (ADR-0015 §Option 2; 0015-research-vllm #40624) found "Gemma4 0% prefix-cache hits when hybrid-attention + DFlash speculative decoding are combined (3 KV-cache specs → block-dropping spiral)," open as of April 2026, with the existing fix (#33524) only guarding 2-group models. For a generic workload that is an annoyance; for us it is fatal, because prefix reuse is the dominant latency lever (ADR-0017 / project_data-first-subcall-layout — we just drove ~6.4x re-prefill toward ~1x). DFlash would zero out exactly the cache hits we engineered for. (This 0% finding was on vLLM; the SGLang DFlash equivalent is untested here, but the mechanism is engine-independent enough to treat as a hard risk.)

Wrong bottleneck regime. Spec-decode accelerates memory-bound LOW-BATCH decode; our workload is prefill-heavy (~314k-char contexts, ADR-0010/0017) and batch-saturated (ADR-0016 GATE #2: co-batching scales to concurrency ~16–24, agg ~680→1461 tok/s). Spec-decode's win shrinks toward zero as the batch fills, and does nothing for prefill.

Feasibility (moot, but also negative). The draft is a GGUF (dflash-draft-3.6-q4_k_m.gguf, ~1.06 GB) while the target is a text-only Gemma4ForCausalLM from a CT/fp8 checkpoint — the target CANNOT be GGUF (GGUF is SGLang's worst-supported path). DFlash plumbing exists in our install (dflash_worker.py, dflash_info.py, model dflash.py, --speculative-dflash-block-size, --speculative-dflash-draft-window-size) so it would load, but a GGUF draft against a CT/fp8 target is an untrodden two-format-dequant path with no evidence. Vocab/tokenizer match is unverified: EAGLE/DFlash drafts must share the target's vocab + tokenizer, and there is no recorded confirmation that dflash-3.6 was trained on the v13 / gemma-4 tokenizer — the "3.6" label does not establish gemma-4 lineage.

Concrete config / values

None to deploy. Leave speculative decoding OFF (SGLang default); do not pass --speculative-algorithm / --speculative-dflash-*.

Revisit ONLY if a future low-concurrency, decode-bound mode appears, under ALL of these gates:

  1. #40624-class hybrid-attention × spec-decode prefix-cache regression confirmed FIXED on the SGLang DFlash path (measure #cached-token with spec on).
  2. Draft tokenizer / vocab proven identical to v13's.
  3. A native (non-GGUF) draft, OR proof SGLang accepts a GGUF draft against a CT/fp8 target without cache/correctness penalty.
  4. Measured acceptance length > ~2 at our concurrency.

Risks & unknowns

  • The 0% figure is a vLLM observation; the SGLang DFlash equivalent is untested (but points the same way).
  • Draft↔target compatibility is entirely unverified.
  • Even a "working" DFlash must BEAT the prefix-cache reuse it disables — an unlikely net win for a prefill-heavy job.

What to measure (only if revisited)

  • #cached-token / #new-token with spec on vs off.
  • Mean accepted draft length at real concurrency (≤1.x ⇒ no benefit).
  • End-to-end s/iter spec on vs off (expect regression while batch-saturated).

Topic 3 — Config Tuning for Our Use-Case

Green — actionable Size everything around ONE invariant: keep total concurrent sub-call KV demand (slots * subcall_context_limit) at roughly HALF the pool so RadixAttention never evicts the orchestrator transcript prefix.

Recommendation

Size everything around one invariant — keep total concurrent sub-call KV demand (slots * subcall_context_limit) at roughly HALF the pool so RadixAttention never evicts the orchestrator transcript prefix. Start from mem-fraction-static 0.80, cuda-graph-max-bs 24, context-length 32768, and a deliberately MODERATE subcall_context_limit (~8k) with concurrency 2, then widen only while prefix reuse stays ≥90%.

Why (grounded in our findings)

The recorded failure (project_sglang-radixattention-pool-tuning, ADR-0016 GATE #2): subcall_context_limit=24000, slots=4 (96k demand vs ~98k pool) → pool 0.40 full, #cached-token~0, ~365s/iter, multihop TIMEOUT — RadixAttention LRU-evicted the orchestrator's growing transcript prefix every iteration, forcing a full re-prefill. Dropping to 8000/slots=2 (16k demand, pool 0.04 full) gave #new-token:153 / #cached-token:3835 = ~96% reuse, ~100s/iter — 3.5x faster. This is the single most important tuning law.

It is distinct from the llama.cpp --kv-unified 500-error problem (ADR-0012) — SGLang evicts gracefully (no 500s) but silently tanks cache + latency. dynamic_kv_pool=True (ADR-0016) removes the ADR-0012 per-slot budget DIVISION (no 500s) — but you STILL want a moderate subcall_context_limit for CACHE LOCALITY (safety and locality are different goals).

mem-fraction interacts with cuda-graph capture (project_int4-quant-v13-w4a16): the real lever is the context-length cap. WITHOUT it, the default ~98k pool grows greedily and starves capture (the old bisection: bs=24 OOMs at 0.78, captures at 0.75). WITH context-length 32768 the SWA pool is bounded to ~96k tokens (~4 GB free), so 0.80 + cuda-graph-max-bs 24 captures the full bs[1..24] set in ~132 s for only 0.05 GB — validated live 2026-06-25. Do NOT use --disable-cuda-graph (keeps only piecewise graphs, kills the decode graph). Keep the hybrid SWA pool — --disable-hybrid-swa-memory is a gemma-4 dead-end (sliding layers 4x wider than full; a uniform pool can't hold both widths; store_cache aborts capture with a shape mismatch — NOT OOM).

Layout / seam interactions. ADR-0017 (data-first {label}:\n{data}\n\n{instruction}) is what MAKES the half-pool reuse achievable — stable context leads, identical chunks match at token 0; instruction-first broke token-0 matching (the ~6.4x re-prefill). ADR-0018 (query-independent extraction MAP) lifted multihop 1/5→5/5 but ENLARGES partials on dense contexts (more reduce tokens, possibly more reduce levels) → higher reduce-step KV pressure → another reason to stay conservative.

Concrete config / values

Server (int4-ct):

--quantization compressed-tensors
--kv-cache-dtype fp8_e4m3
--attention-backend triton
--mem-fraction-static 0.80
--cuda-graph-max-bs 24
--context-length 32768
--tp 1
--enable-metrics

Client / harness:

slots (MAPREDUCE_CONCURRENCY) = 2
dynamic_kv_pool              = True
subcall_base_url            = None        # one endpoint
subcall_context_limit       ~ 8000        # slots*limit = 16k vs ~98k pool
                                          # -> ~0.04-0.16 full, room for the
                                          # resident ~20-40k orchestrator prefix
recommended_chunk_chars     ~ 30% of per-call budget  # ADR-0010

Tuning order (one variable at a time, watch #cached-token):

  1. Land data-first layout (ADR-0017) + extraction MAP (ADR-0018) FIRST — without them no config makes the cache hit.
  2. Fix mem-fraction-static 0.80 + cuda-graph-max-bs 24 + context-length 32768; confirm clean capture (full bs[1..24] set, ~0.05 GB).
  3. Set slots=2, subcall_context_limit=8000; confirm #cached-token >> #new-token (≥90%) and ~100s/iter.
  4. ONLY THEN widen for throughput, one step at a time, holding slots * subcall_context_limit ≤ 0.5 * max_total_num_tokens; back off the moment reuse drops or s/iter jumps (the eviction knee).
  5. context-length 32768 fits the per-sub-call need; raise only if a single (now-fatter) REDUCE needs it — it costs pool, so re-check the budget.

Risks & unknowns

  • Tuning fixed LATENCY 3.5x but NOT accuracy by itself — config is necessary, not sufficient.
  • ADR-0018 larger partials may push reduce KV past the conservative budget on dense contexts (watch reduce-level count; one-line revertible _SEAM_EXTRACTION_MAP=False).
  • The half-pool rule is anchored on a 2–5 task sample; the "≤0.5×pool" coefficient may need re-derivation with bf16 KV (smaller pool).
  • If the root orchestrator prefix alone exceeds its reserve on very long solves, transcript compaction is a separate still-open lever (ADR-0012 §open).

What to measure

  • Per-Prefill-batch #cached-token vs #new-token (target ≥90%; ignore the cache_hit_rate gauge).
  • Pool occupancy fraction (keep well under 0.5 under concurrency).
  • s/iter on multihop (~100s target) and task completion / timeout rate (multihop_053 was the persistent timeout).
  • cuda-graph capture success at the chosen mem-fraction / bs.

Cross-Cutting Priorities

  1. Run the GATE #2 accuracy A/B before flipping any default (ADR-0016 stays PROPOSED). The stack's gate is accuracy (~0 correct on small samples). Disentangle the three suspects — (a) fp8 KV precision, (b) chat-template / reasoning_parser=gemma4 thinking-mode, (c) small-sample variance — against the llama q8_0 baseline, more tasks, memory on.
  2. Isolate KV dtype on int4-ct weights: fp8_e4m3 vs bf16 KV. int4-ct weights (~8.5 GB) decouple KV dtype from pool pressure for the first time — the cleanest, cheapest experiment to kill or confirm the biggest accuracy suspect.
  3. Lock data-first layout (ADR-0017) + extraction MAP (ADR-0018) as prerequisites, then tune to the half-pool rule. Together these produce the ~96% reuse / 3.5x-faster / 5-of-5-multihop regime; tuning on an instruction-first layout or a per-query MAP is wasted.
  4. Speculative decoding stays OFF — close DFlash as NO-GO. It disables our dominant lever (prefix reuse, #40624) and targets a regime (low-batch decode) we never enter.
  5. TurboQuant stays iceboxed; pre-quantized fp8/AWQ-on-disk is the only KV-adjacent follow-up worth carrying (faster start + more headroom, orthogonal to dtype). Revisit 4-bit KV only if a future roadmap is genuinely capacity-bound.
